结合当前项目文档，建议把这件事拆成一个**“输入源接入与解析能力方案”**，目标不是一开始就把 Git、缺陷、流量平台全部真实打通，而是先把**来源限制、状态流转、统一数据模型、解析流程、可复用开源能力**设计清楚。这样 Phase 1 可以先落骨架，Phase 2 再逐步接真实数据源。

现有 Phase 1 的目标本来就是先完成 4 页签结构、输入材料页签基础结构、覆盖与风险/执行与回流骨架，并为后续真实接口和分析能力预留结构；真实 GitLab/GitHub、缺陷平台、流量平台对接暂不纳入 Phase 1。 这和当前方案很吻合：**先做统一输入源模型和状态体系，后续再接外部平台。**

---

# 1. 方案目标

本方案要解决 4 个问题：

1. **来源怎么限制**：不同来源允许什么输入方式、什么文件格式、什么数据结构。
2. **解析怎么流转**：文件上传、文本输入、外部接入之后，如何从“待接入”走到“已解析 / 解析失败”。
3. **外部来源怎么接**：接口文档、代码变更、缺陷单、流量摘要分别怎么接入，Phase 1 和 Phase 2 怎么分层。
4. **哪些开源能力可以复用**：文档解析、文件上传、状态机、表单、表格等尽量不要从 0 写。

最终要支撑文档里定义的输入材料页目标：集中管理 AI 生成依赖的输入证据，让用户清楚知道“本次生成用了什么”，并支持用户选择是否参与本次生成。

---

# 2. 总体架构设计

建议拆成 6 层：

```text
用户输入层
  ↓
来源接入层
  ↓
解析处理层
  ↓
统一来源模型层
  ↓
生成控制层
  ↓
AI 生成与分析层
```

---

## 2.1 用户输入层

负责页面交互。

包括：

* 需求描述 / PRD 文本输入
* 附件上传
* 接口文档导入
* 代码变更关联
* 缺陷单关联
* 流量摘要选择

这一层只做**采集和展示**，不要在前端承担复杂解析逻辑。

---

## 2.2 来源接入层

负责把不同来源转成统一的输入源对象。

来源包括：

| 来源   | 接入方式                            | Phase 1 建议 | Phase 2 建议      |
| ---- | ------------------------------- | ---------- | --------------- |
| 需求描述 | 文本框输入 / 粘贴 PRD                  | 直接支持       | 增加结构化抽取         |
| 附件材料 | 上传 PDF / DOCX / TXT / PNG / JPG | 复用现有附件上传   | 增加解析结果          |
| 接口文档 | OpenAPI / Swagger / URL / 文本粘贴  | 占位 + 手动输入  | 接真实导入           |
| 代码变更 | PR / MR / Commit / Diff 摘要      | 占位 + 手动摘要  | 接 GitLab/GitHub |
| 缺陷单  | 缺陷 ID / 批量导入 / 平台选择             | 占位 + 手动摘要  | 接缺陷平台           |
| 流量摘要 | 时间范围 / 接口热点 / 异常摘要              | 占位 + 手动摘要  | 接流量平台           |

你们接口文档里也已经提出了输入材料结构，包括 `requirementText`、`interfaceDocs`、`attachments`、`codeChanges`、`defects`、`trafficSummaries`、`selectedSourceTypes`。 所以这里不要重新发明一套，而是把它扩展成更完整的 `InputSourceItem`。

---

## 2.3 解析处理层

负责把原始材料转换成 AI 可用文本和结构化片段。

主要任务：

* 文件类型识别
* 文本提取
* Markdown / HTML 清洗
* 图片 OCR，Phase 1 可不做
* OpenAPI 结构化解析
* Diff / Commit 摘要抽取
* 缺陷单字段归一化
* 流量摘要归一化
* 去重、截断、分块
* 解析失败原因记录

---

## 2.4 统一来源模型层

所有来源最后都要落成同一个模型。

建议模型叫：

```ts
type SourceType =
  | 'requirement'
  | 'attachment'
  | 'interface_doc'
  | 'code_change'
  | 'defect'
  | 'traffic_summary';

type ContentStatus =
  | 'empty'
  | 'filled'
  | 'pending_connection'
  | 'connected';

type ParseStatus =
  | 'not_required'
  | 'pending'
  | 'parsing'
  | 'parsed'
  | 'failed';

type ParticipateStatus =
  | 'included'
  | 'excluded';

interface AiInputSourceItem {
  id: string;
  workspaceId: string;

  type: SourceType;
  title: string;
  description?: string;

  rawContent?: string;
  parsedContent?: string;

  contentStatus: ContentStatus;
  parseStatus: ParseStatus;
  participateStatus: ParticipateStatus;

  sourceRef?: {
    platform?: 'local' | 'openapi' | 'gitlab' | 'github' | 'jira' | 'zentao' | 'traffic';
    refId?: string;
    url?: string;
  };

  metadata?: {
    fileName?: string;
    fileType?: string;
    fileSize?: number;
    tokenEstimate?: number;
    itemCount?: number;
    createdAt?: string;
    updatedAt?: string;
    parseErrorMessage?: string;
  };
}
```

---

## 2.5 生成控制层

负责决定“能不能生成”和“用哪些材料生成”。

这一层不要写死在 UI 里，建议前后端都保持一致规则。

核心判断：

```ts
interface GenerationReadiness {
  canGenerate: boolean;
  level: 'blocked' | 'ready' | 'suggest_improve';
  includedSources: AiInputSourceItem[];
  missingSuggestions: string[];
  blockedReasons: string[];
}
```

生成规则建议：

| 场景                  | 生成状态      | 说明          |
| ------------------- | --------- | ----------- |
| 没有任何来源参与            | 不可生成      | 按钮置灰        |
| 只有需求文本参与            | 可生成，但建议补充 | 可以生成首版      |
| 需求 + 附件 / 接口 / 缺陷参与 | 推荐生成      | 信息更完整       |
| 参与来源正在解析            | 建议等待      | 可选择跳过解析中的来源 |
| 参与来源解析失败            | 不纳入生成     | 提示处理失败来源    |

负责把输入源组装成 prompt / payload，然后调用现有生成接口。

你们当前已有 `POST /api/ai-cases/generate` 和 `POST /api/ai-cases/generate/stream`，可以继续复用。

      "title": "历史登录缺陷",

# 3. 来源限制设计

建议你在产品和代码里都定义一份**来源限制配置表**，前端根据它渲染 UI，后端根据它校验。

---

## 3.1 来源限制总表

```ts
interface SourceTypeConfig {
  type: SourceType;
  label: string;
  allowedInputModes: Array<'text' | 'file' | 'url' | 'platform_ref'>;
  allowedFileTypes?: string[];
  maxFileSizeMB?: number;
  maxCount?: number;
  requireParsing: boolean;
  canParticipateInGeneration: boolean;
}
```

---

## 3.2 建议限制

| 来源   | 允许输入方式                | 文件限制                    | 数量限制              | 是否需要解析  | 是否可参与生成 |
| ---- | --------------------- | ----------------------- | ----------------- | ------- | ------- |
| 需求描述 | 文本                    | 无                       | 1                 | 否 / 轻解析 | 是       |
| 附件材料 | 文件上传                  | PDF、DOCX、TXT、MD、PNG、JPG | 建议 10 个以内         | 是       | 是       |
| 接口文档 | 文件 / URL / 文本         | JSON、YAML、TXT           | 建议 5 个以内          | 是       | 是       |
| 代码变更 | 平台引用 / 文本摘要 / Diff 文件 | DIFF、PATCH、TXT          | 建议 20 个 commit 以内 | 是       | 是       |
| 缺陷单  | 平台引用 / ID / CSV / 文本  | CSV、TXT                 | 建议 50 条以内         | 是       | 是       |
| 流量摘要 | 平台引用 / 文本摘要 / CSV     | CSV、JSON、TXT            | 建议按时间窗口           | 是       | 是       |

---

## 3.3 为什么要限制

因为 AI 输入不是越多越好，需要控制：

* token 成本
* 解析耗时
* 噪音来源
* 用户误传无关材料
* 生成结果偏离需求

所以每个来源都应该有：

* 最大数量
* 最大大小
* 支持格式
* 超限提示
* 解析失败处理

# 4. 解析状态流转设计

你前面提到的状态标签方向是对的，但建议落成状态机。

---

## 4.1 状态分三类

### 内容状态 contentStatus

```ts
type ContentStatus =
  | 'empty'              // 未填写
  | 'filled'             // 已填写
  | 'pending_connection' // 待接入
  | 'connected';         // 已接入
```

### 解析状态 parseStatus

```ts
type ParseStatus =
  | 'not_required' // 不需要解析
  | 'pending'      // 待解析
  | 'parsing'      // 解析中
  | 'parsed'       // 已解析
  | 'failed';      // 解析失败
```

### 参与状态 participateStatus

```ts
type ParticipateStatus =
  | 'included' // 已参与
  | 'excluded'; // 未参与
```

---

## 4.2 推荐状态流转

```text
待接入
  ↓
已接入
  ↓
待解析
  ↓
解析中
  ↓
已解析 ──→ 已参与 / 未参与
  ↓
解析失败 ──→ 重新解析 / 删除 / 查看原因
```

需求描述比较特殊，可以简化：

```text
未填写 → 已填写 → 已参与 / 未参与
```

---

## 4.3 前端按钮状态

| 状态        | 用户可做什么           |
| --------- | ---------------- |
| 未填写       | 填写需求             |
| 已填写       | 编辑 / 清空 / 参与生成   |
| 待接入       | 接入来源             |
| 已接入 + 待解析 | 开始解析 / 等待自动解析    |
| 解析中       | 查看进度 / 取消，非必需    |
| 已解析       | 查看结果 / 参与生成 / 删除 |
| 解析失败      | 查看原因 / 重试 / 删除   |

# 5. 接入逻辑设计

---

## 5.1 需求描述 / PRD

这是最基础来源。

### 输入方式

* 文本框直接输入
* 粘贴 PRD 片段
* 从模板填充
* 从附件解析后回填摘要，Phase 2 可做

### 处理逻辑

```text
用户输入文本
  ↓
前端自动保存
  ↓
更新 requirementText
  ↓
contentStatus = filled
parseStatus = not_required / parsed
participateStatus = included
```

### 说明

需求描述不一定需要复杂解析。Phase 1 可以视为 `parsed`，Phase 2 再做结构化抽取：

* 背景
* 目标
* 主流程
* 异常流程
* 验收标准
* 业务规则

---

## 5.2 附件材料

附件是你当前最容易落地的增强输入，因为你们已有附件接口。文档中也明确附件能力是 Phase 1 需要保留和复用的主能力之一。

### 支持格式

建议：

* PDF
* DOCX
* TXT
* MD
* PNG / JPG，Phase 1 可以只上传不 OCR
* XLSX，暂不建议第一期支持，解析复杂度高

### 处理逻辑

```text
用户上传附件
  ↓
POST /attachments
  ↓
创建 SourceItem
  ↓
contentStatus = connected
parseStatus = pending
  ↓
后端解析任务
  ↓
parsed / failed
  ↓
用户选择是否参与生成
```

### 解析方式

* TXT / MD：直接读取文本
* DOCX：Mammoth 或 Tika
* PDF：Tika 或 PDF.js
* PNG / JPG：Phase 1 先作为附件证据，不参与文本生成；Phase 2 接 OCR

Apache Tika 可以检测并提取大量文件类型的元数据和文本，适合后端统一解析入口；PDF.js 是基于 Web 标准的 PDF 解析和渲染平台，适合前端预览或轻量提取；Mammoth 专注把 DOCX 转为干净的 HTML，适合处理 Word 文档。([tika.apache.org][1])

---

## 5.3 接口文档

### 输入方式

* OpenAPI JSON / YAML 文件上传
* Swagger URL 导入
* 文本粘贴
* 手动摘要，Phase 1 可用

### 处理逻辑

```text
导入 OpenAPI / Swagger
  ↓
校验格式
  ↓
提取 path / method / parameters / response / auth
  ↓
结构化成 InterfaceDocSource
  ↓
参与生成
```

### 结构化结果建议

```ts
interface ParsedInterfaceDoc {
  endpoints: Array<{
    path: string;
    method: string;
    summary?: string;
    parameters?: unknown[];
    requestBody?: unknown;
    responses?: unknown;
    authRequired?: boolean;
  }>;
}
```

### 对生成的价值

接口文档主要帮助生成：

* 接口覆盖
* 参数校验
* 状态码测试
* 鉴权测试
* 边界值测试
* 幂等性 / 超时 / 重试类测试

---

## 5.4 代码变更

Phase 1 不建议真实接 GitLab / GitHub，因为文档里已明确暂不纳入。
但是页面和模型可以先预留。

### Phase 1 输入方式

* 用户粘贴 Diff 摘要

* GitLab MR

### 处理逻辑

```text

提取变更文件、函数、接口、模块

AI / 规则生成变更摘要

interface ParsedCodeChange {

```

# 6. 开源能力选型建议

这里按你当前项目的落地难度来推荐。

---

## 6.1 文件上传

### 推荐

* Uppy
* react-dropzone

Uppy 是模块化开源 JavaScript 文件上传器，并提供 React 组件和 UI 插件；这类能力适合你们附件上传、拖拽、进度、失败重试等场景。([uppy.io][2])

### 建议

如果你们现在已经有附件上传能力，不一定要重构。可以：

* 保留现有上传接口
* 前端上传 UI 可逐步替换为 Uppy / react-dropzone
* 解析状态和 SourceItem 绑定到现有附件记录

---

## 6.2 文档解析

### 后端优先

* Apache Tika

适合统一处理：

* PDF
* DOCX
* TXT
* PPT
* Excel，后期
* 多种文档元数据

Tika 的优势是后端统一入口，不用前端分别处理各种格式。官方说明它能检测并提取超过千种文件类型的元数据和文本。([tika.apache.org][1])

### 前端辅助

* PDF.js：适合 PDF 预览和前端渲染。([mozilla.github.io][3])
* Mammoth：适合 DOCX 转 HTML 或抽取语义结构。([GitHub][4])

### 推荐落地

Phase 1：

* TXT / MD 前端或后端直接读
* PDF / DOCX 只上传展示，不强制解析
* 解析状态可以先 mock 或异步占位

Phase 2：

* 后端接 Tika 统一解析
* 前端只展示解析状态和解析结果预览

---

## 6.3 状态流转

### 推荐

* XState
* 或轻量自定义状态机

XState 是 JavaScript / TypeScript 的状态管理与编排方案，适合表达事件驱动、状态机、复杂状态流转；官方也将其定位为 JS/TS 的状态管理与编排解决方案。([stately.ai][5])

### 为什么适合你这里

输入源状态不是简单枚举，而是有明确流转：

```text
待接入 → 已接入 → 待解析 → 解析中 → 已解析 / 解析失败 → 已参与
```

用状态机可以避免：

* 解析失败还能参与生成
* 未接入却显示已解析
* 解析中被误删没有提示
* 已参与来源其实无内容

---

## 6.4 前端状态管理

### 推荐

* Zustand
* Redux Toolkit

建议你们如果项目已经有状态管理，就不要新增复杂依赖。
这里更关键的是把状态分清：

```ts
interface InputSourcesStore {
  sources: AiInputSourceItem[];
  selectedSourceIds: string[];
  activeSourceType?: SourceType;
  readiness: GenerationReadiness;
}
```

---

## 6.5 表单管理

### 推荐

* React Hook Form

适合：

* 新增需求抽屉
* 需求描述编辑
* 外部来源配置表单
* 缺陷单批量导入表单

---

## 6.6 表格与列表

### 推荐

* TanStack Table

适合：

* 全部记录
* 附件列表
* 缺陷单列表
* 接口列表

# 7. 接口方案设计

你们接口清单已经给了比较明确的方向：优先扩展工作台详情接口，再新增输入材料更新接口、覆盖风险接口、执行摘要接口等。

# 9. UI 设计方案

---

## 9.1 Tips 设计

你提到的“阶段说明”不建议长期裸露在 tab 下方。
更好的方式是：

```text
[1 输入材料 ⓘ] [2 生成结果 ⓘ] [3 覆盖与风险 ⓘ] [4 执行与回流 ⓘ]
```

hover 展示 tooltip。

### 文案建议

| 阶段    | Tips 文案                               |
| ----- | ------------------------------------- |
| 输入材料  | 准备本次生成依赖的材料，并确认参与范围。支持文本、附件与外部来源接入。   |
| 生成结果  | 查看 AI 生成的结构化用例结果，可进行筛选、编辑、补充与确认。      |
| 覆盖与风险 | 评估高风险点、覆盖缺口与推荐补充项，辅助判断是否需要补充测试场景。     |
| 执行与回流 | 发布工作台，触发执行并查看结果；将执行反馈、质量评分和知识沉淀回流到平台。 |

---

## 9.2 来源卡设计

每个来源卡统一展示：

```text
[图标] 来源名称
一句话说明

内容状态：[已接入]
解析状态：[已解析]
参与状态：[已参与] / 开关

[主操作]
```

建议卡片状态优先级：

1. 解析失败，红色最高优先
2. 解析中，蓝色 loading
3. 待接入，橙色
4. 已解析且参与，绿色
5. 未参与，灰色

---

## 9.3 右侧生成准备卡

右侧不要只是摘要，应该是决策区：

```text
生成准备情况

可开始生成

本次参与来源：
✓ 需求描述 已参与
✓ 附件材料 已参与（2）
✓ 接口文档 已参与（1）
－ 代码变更 未参与

缺失与建议：
- 建议补充代码变更，可增强改动风险识别
- 流量摘要未接入，热点场景覆盖可能不足

[保存草稿] [AI 生成首版用例]
```

# 10. 研发落地计划

结合你们 Phase 1 排期，两周内建议这样切。

---

## Phase 1：页面和状态骨架

目标：页面先跑起来，来源模型先统一。

范围：

* 输入材料页 UI
* Tips tooltip
* 来源限制配置
* 状态标签统一
* 附件区复用
* 外部来源占位
* 本次参与来源选择
* 生成准备情况卡
* 生成按钮启停逻辑

对应你们排期里的 C1 输入材料页签容器、C2 需求描述区迁移、C3 附件区复用、C4 外部来源占位区。

---

## Phase 2：解析与真实来源接入

目标：让输入源真的被 AI 使用。

范围：

* 文件解析服务
* 接口文档解析
* 缺陷平台接入
* Git 变更摘要
* 生成接口支持多 sourceIds
* 生成结果增加来源依据

这也符合你们改造大纲中 Phase 2“输入源与生成增强”的定位：引入输入材料模型、对接 Git 变更摘要、接口文档导入，并为生成结果增加来源说明。

---

## Phase 3：风险分析与执行闭环

目标：从“多源生成”进入“风险驱动”和“执行回流”。

范围：

* 风险评分
* 覆盖缺口分析
* 推荐补充项
* Jenkins 执行联动
* 质量评分与知识回流

你们大纲也将风险评分、覆盖缺口、推荐补充、Jenkins 执行联动和质量回流放在 Phase 3。

# 13. 最终推荐方案

我建议你按照这个优先级推进：

```text
第一优先级：
统一来源模型 + 状态体系 + UI 展示

第二优先级：
来源限制配置 + 生成准备判断 + 按来源参与生成

第三优先级：
附件解析能力 + 接口文档解析

第四优先级：
Git / 缺陷 / 流量真实平台接入

第五优先级：
风险分析、覆盖分析、执行回流
```

这样不会一下子陷入“所有平台都要接”的复杂度里，而是先把工作台的产品骨架、数据模型和状态逻辑打稳。等 Phase 2 开始，真实来源接入就只是往统一模型里填数据，而不是每接一个平台就重做一套页面和逻辑。
