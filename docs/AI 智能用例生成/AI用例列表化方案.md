# AI 用例列表化方案

## 1. 背景判断

当前 AI 工作台虽然已经具备脑图能力，但从产品使用价值看，脑图并不是高频核心能力：

- 真实高频动作是审阅、筛选、批量修改、分派、执行跟踪
- 脑图更适合发散式设计，不适合持续运营型工作台
- 引入 `mind-elixir` 会让结果页长期背负画布、节点、布局、主题和交互维护成本
- 后续还要接风险分析、执行回流、质量评分，这些天然更适合结构化列表

因此建议把 AI 工作台从“脑图生成器”明确转成“结构化测试用例工作台”。

## 2. 产品定位调整

新的定位建议写成：

> AI 工作台是一个基于多源输入生成、审阅、补充、执行和回流结构化测试用例的工作台。

关键词建议改成：

- 结构化
- 可审阅
- 可筛选
- 可执行
- 可回流

不再强调：

- 脑图生成
- 节点编辑
- 画布式操作

## 3. 产品形态建议

建议将“生成结果”页彻底改为 `列表优先`，并按两层结构设计：

### 3.1 第一层：列表总览

用于快速浏览和批量判断：

- 总数
- 风险分布
- 状态分布
- 模块分布
- 来源分布
- 筛选后的用例列表

### 3.2 第二层：详情抽屉

用于单条用例的深度查看和编辑：

- 标题
- 所属模块
- 前置条件
- 步骤
- 预期结果
- 风险级别
- 生成依据
- 执行状态
- 负责人
- 备注 / 回流信息

也就是说，不再依赖“左边脑图 + 右边详情”的模式，而改成：

- 上方摘要
- 中间筛选
- 下方列表
- 侧边抽屉详情

## 4. 页面风格建议

这个页面不建议做成“卡片瀑布流”，更适合做成“运营后台式结构化工作台”。

### 4.1 风格关键词

- 干净
- 稳定
- 密度适中
- 便于扫读
- 以数据为中心

### 4.2 推荐视觉方向

- 浅色背景 + 白色面板
- 摘要区数字更强，正文弱化
- 风险标签用暖色
- 状态标签用中性色 / 绿色 / 红色
- 表格行高适中，不要做得太松
- 详情抽屉强调字段分组，而不是大段自由文本

### 4.3 不建议的风格

- 大面积渐变
- 过多悬浮卡片
- 伪大屏科技感
- 过于花哨的 AI 风格装饰

## 5. 结果列表页信息架构

推荐结构如下：

1. 顶部工作台头部
2. 统计摘要区
3. 快速筛选区
4. 结构化用例列表
5. 右侧详情抽屉

## 6. 页面区块设计

### 6.1 顶部头部

建议展示：

- 工作台名称
- 所属项目 / 模块
- 当前状态
- 版本号
- 最近更新时间
- 发布按钮
- 执行按钮

### 6.2 统计摘要区

建议保留 6 张摘要卡：

- 用例总数
- 高风险数
- 待补充数
- 已覆盖率
- 最近执行状态
- 质量评分

### 6.3 快速筛选区

建议做成一行高频筛选器：

- 模块
- 优先级
- 风险级别
- 来源类型
- 执行状态
- 负责人
- 是否仅看未覆盖
- 搜索框

### 6.4 列表主体

列表建议优先支持：

- 单选查看详情
- 多选批量操作
- 固定表头
- 列排序
- 风险和状态颜色标识
- 行内快速操作

### 6.5 详情抽屉

打开单条用例后，建议右侧抽屉分成 5 个分组：

1. 基本信息
2. 测试内容
3. 风险与来源
4. 执行与回流
5. 备注与附件

## 7. AI 用例字段设计

这里按“列表字段”和“详情字段”分开设计。

### 7.1 列表字段

建议列表页默认列如下：

| 字段 | 含义 | 是否默认展示 |
|---|---|---|
| `caseId` | 用例唯一标识 | 是 |
| `title` | 用例标题 | 是 |
| `moduleName` | 所属模块 | 是 |
| `caseType` | 用例类型 | 是 |
| `priority` | 优先级 | 是 |
| `riskLevel` | 风险等级 | 是 |
| `coverageStatus` | 覆盖状态 | 是 |
| `executionStatus` | 执行状态 | 是 |
| `sourceSummary` | 来源摘要 | 是 |
| `owner` | 负责人 | 是 |
| `updatedAt` | 最近更新时间 | 是 |
| `qualityScore` | 质量分 | 否 |
| `lastExecutedAt` | 最近执行时间 | 否 |
| `tags` | 标签 | 否 |

### 7.2 详情字段

建议详情抽屉字段如下：

| 字段 | 含义 |
|---|---|
| `caseId` | 用例 ID |
| `title` | 用例标题 |
| `moduleName` | 所属模块 |
| `featureName` | 所属功能点 |
| `caseType` | 功能 / 接口 / 异常 / 回归 / 冒烟 |
| `priority` | P0 / P1 / P2 / P3 |
| `riskLevel` | high / medium / low |
| `coverageStatus` | covered / partial / uncovered |
| `executionStatus` | todo / ready / running / passed / failed / blocked / skipped |
| `preconditions` | 前置条件 |
| `steps` | 测试步骤 |
| `expectedResults` | 预期结果 |
| `testData` | 测试数据 |
| `evidences` | 生成依据列表 |
| `riskReason` | 风险说明 |
| `recommendationReason` | AI 推荐原因 |
| `owner` | 负责人 |
| `reviewStatus` | 审核状态 |
| `lastExecutionId` | 最近执行记录 ID |
| `lastExecutedAt` | 最近执行时间 |
| `lastExecutionResult` | 最近执行结果摘要 |
| `qualityScore` | 质量评分 |
| `knowledgeBaseStatus` | 是否回流知识库 |
| `remarks` | 备注 |
| `attachments` | 附件列表 |

### 7.3 推荐的 TypeScript 结构

```ts
type AiCaseType = 'functional' | 'api' | 'exception' | 'regression' | 'smoke';

type AiRiskLevel = 'high' | 'medium' | 'low';

type AiCoverageStatus = 'covered' | 'partial' | 'uncovered';

type AiExecutionStatus =
  | 'todo'
  | 'ready'
  | 'running'
  | 'passed'
  | 'failed'
  | 'blocked'
  | 'skipped';

type AiSourceType =
  | 'requirement'
  | 'interface_doc'
  | 'attachment'
  | 'code_change'
  | 'defect'
  | 'traffic_summary'
  | 'knowledge_base';

interface AiCaseEvidence {
  id: string;
  sourceType: AiSourceType;
  sourceLabel: string;
  sourceRef?: string | null;
  summary?: string | null;
}

interface AiCaseListItem {
  caseId: string;
  title: string;
  moduleName: string;
  featureName?: string | null;
  caseType: AiCaseType;
  priority: 'P0' | 'P1' | 'P2' | 'P3';
  riskLevel: AiRiskLevel;
  coverageStatus: AiCoverageStatus;
  executionStatus: AiExecutionStatus;
  sourceSummary: string;
  owner?: string | null;
  updatedAt: string;
  qualityScore?: number | null;
  lastExecutedAt?: string | null;
  tags?: string[];
}

interface AiCaseDetail extends AiCaseListItem {
  preconditions: string[];
  steps: string[];
  expectedResults: string[];
  testData?: string[];
  evidences: AiCaseEvidence[];
  riskReason?: string | null;
  recommendationReason?: string | null;
  reviewStatus?: 'draft' | 'reviewing' | 'approved' | 'rejected';
  lastExecutionId?: number | null;
  lastExecutionResult?: string | null;
  knowledgeBaseStatus?: 'none' | 'candidate' | 'added';
  remarks?: string | null;
  attachments?: Array<{
    id: string;
    name: string;
    url?: string | null;
  }>;
}
```

## 8. 列表默认列建议

如果首屏列太多，用户会很累。建议默认只展示这 10 列：

1. 用例标题
2. 模块
3. 类型
4. 优先级
5. 风险等级
6. 覆盖状态
7. 执行状态
8. 来源
9. 负责人
10. 更新时间

其他列放到：

- 列设置
- 详情抽屉
- hover 提示

## 9. 列表交互建议

### 9.1 行级操作

每行建议保留这几个高频操作：

- 查看详情
- 标记优先级
- 变更状态
- 指派负责人
- 加入执行范围

### 9.2 批量操作

支持批量：

- 修改优先级
- 修改负责人
- 标记为已覆盖
- 加入执行任务
- 加标签

### 9.3 详情交互

点击整行打开右侧抽屉，而不是跳新页。

抽屉里重点支持：

- 查看 AI 生成依据
- 复制步骤
- 快速编辑测试内容
- 查看最近执行记录

## 11. 去脑图的技术建议

如果产品方向确认彻底去脑图，建议分两步做。

### 11.1 第一步：列表化切换

先做：

- 默认结果页切到列表视图
- 页面文案改成“列表优先”
- 新功能只围绕列表页建设
- 脑图仅保留兼容入口

### 11.2 第二步：彻底卸载脑图

确认没人再使用脑图后，再做：

- 删除 `mind-elixir` 依赖
- 清理 `AICases.tsx` 里的画布逻辑
- 清理 `src/lib/aiCaseMindMap.ts`
- 清理 `src/types/aiCases.ts` 中与 `MindElixirData` 强绑定的结构
- 把结果数据主结构从 `mapData` 迁移到 `cases[]`

## 12. 推荐落地顺序

建议按这个顺序推进：

1. 默认视图改成列表
2. 明确列表字段和状态枚举
3. 把结果页做成“筛选 + 表格 + 抽屉”
4. 再逐步弱化脑图入口
5. 最后彻底移除脑图依赖

## 13. 结论

如果当前“几乎没人用脑图”这个判断成立，那从产品和工程两个角度看，转成结构化列表都是更合适的路线。

因为你们真正要做的不是“把测试点画出来”，而是：

- 让 AI 用例能被审阅
- 让 AI 用例能被筛选
- 让 AI 用例能被执行
- 让 AI 用例能被回流

这些能力的最佳载体，本质上就是列表，而不是脑图。
