# AI 工作台列表版 UI 草图

## 1. 目标

这份草图用于把 AI 工作台从“脑图编辑器”重构为“结构化用例工作台”。

设计重点：

- 首页不空
- 列表页可审阅
- 抽屉页可编辑
- 后续风险、执行、回流能力可平滑扩展

## 2. 页面总结构

建议整体分成两层：

1. `AI 工作台 / 工作台首页`
2. `AI 工作台 / 全部记录`
3. `AI 工作台详情页`

其中：

- 首页负责进入、引导、继续历史工作台
- 全部记录页负责搜索、筛选、分页和管理历史工作台
- 详情页负责输入、结果、风险、执行的具体工作

## 4. 详情页总布局

详情页建议不再使用“脑图画布 + 浮动侧栏”的模式，而改成：

- 头部信息区
- 摘要卡区
- 页签区
- 当前页签主体

建议仍保留 4 个页签：

1. 输入材料
2. 生成结果
3. 覆盖与风险
4. 执行与回流

这样产品结构不变，但交互载体从脑图切成列表。

## 9. 需要提前预留的扩展点

### 9.1 表格列配置

不要把列写死在 JSX 里，建议做成列配置数组：

- 方便 Phase 2 增加来源、执行、评分列
- 方便后续接列设置
- 方便根据权限隐藏列

### 9.2 筛选器配置

筛选器建议也做成 schema：

- `key`
- `label`
- `type`
- `options`
- `defaultValue`

这样后面接更多来源或状态时，不用重写筛选区。

### 9.3 详情抽屉分区

详情抽屉建议固定为可扩展的 section：

- 基本信息
- 测试内容
- 来源与风险
- 执行与回流
- 备注与附件

未来只是在 section 里加字段，不改抽屉骨架。

### 9.4 页签数据适配层

前端建议把数据访问隔开：

- 页面组件不直接处理后端原始字段
- 通过 adapter 转成列表页消费的 view model

这样后面从 `mapData` 迁移到 `cases[]` 时，页面可以少改很多。

## 10. 视觉方向建议

建议维持：

- 浅背景
- 中高密度
- 表格为中心
- 暖色强调风险
- 静态白卡承载信息

不建议：

- 做成满屏大卡片
- 用很重的渐变科技感
- 用很大的留白稀释信息密度

## 11. 结论

UI 草图的核心思路就是：

- 首页像“入口页”
- 全部记录页像“历史管理页”
- 结果页像“运营型工作台”
- 风险页像“分析页”
- 执行页像“回流页”

这套结构对后续功能扩展最稳，也最适合替换掉脑图形态。
