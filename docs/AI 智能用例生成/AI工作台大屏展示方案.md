# AI 工作台大屏展示方案

## 1. 目标

基于当前 `AICases` 四页签工作台骨架，补出一个更适合“汇总查看、方案评审、状态跟踪”的大屏展示方案。

这里先明确一个核心判断：

- 当前四页签结构适合“编辑型工作台”
- 大屏展示页更适合“驾驶舱型总览”

也就是说，大屏不应该只是把现有编辑页放大，而应该把输入、生成、风险、执行四个阶段压缩成一屏可扫读的信息面板。

## 2. 当前页面的问题

结合现有截图和 `src/pages/cases/AICases.tsx` 的实现，当前页面已经有：

- 顶部摘要卡
- 四个工作台页签
- 生成结果的脑图 / 列表双视图
- 覆盖与风险、执行与回流的骨架

但如果把它当“大屏展示页”，还存在几个明显问题：

1. 信息虽然分区了，但仍偏“操作型页面”，不是“汇报型页面”
2. 关键判断信息不够集中，用户需要切页签才能知道工作台是否健康
3. 没有突出“下一步该做什么”
4. 空态过于稀疏，页面视觉重心不足
5. 脑图适合深度编辑，不适合作为大屏首屏核心内容

## 3. 设计建议

建议把 AI 工作台拆成两层：

### 3.1 第一层：工作台总览大屏

用于展示：

- 当前工作台是否可生成
- 本次生成基于哪些材料
- 生成结果规模如何
- 风险和覆盖是否达标
- 是否已发布、已执行、可回流

适合场景：

- 日常跟踪
- 周会 / 评审会展示
- QA 负责人快速判断
- 工作台空态和半完成态引导

### 3.2 第二层：四页签编辑工作台

用于实际操作：

- 输入材料维护
- 脑图编辑
- 风险补充
- 执行与回流操作

适合场景：

- 编辑测试点
- 修正生成结果
- 深入查看风险项
- 执行和回填

## 4. 大屏应该展示哪些信息

建议按“全局概览 > 当前结果 > 风险判断 > 下一步动作”的顺序组织。

### 4.1 全局信息

放在页面最上方，作为统一头部：

- 工作台名称
- 所属项目 / 模块
- 当前状态
- 最近更新时间
- 当前版本
- 发布状态
- 远端同步状态
- 主操作按钮

建议状态统一成少量、可感知的枚举：

- 草稿中
- 待生成
- 生成中
- 待评审
- 待执行
- 已执行
- 可回流

### 4.2 KPI 摘要

这是大屏第一眼必须看到的区域，建议保留 5 到 6 个指标：

- 输入材料数
- 生成用例数
- 高风险项数
- 覆盖率
- 最近执行状态
- 质量评分

其中最关键的是这 3 个：

- 高风险项数
- 覆盖率
- 最近执行状态

### 4.3 输入材料概览

这一块不需要展开编辑表单，而是展示“这次生成用了什么证据”：

- 需求文本是否已准备
- 接口文档数量
- 附件数量
- 代码变更来源
- 缺陷来源
- 流量摘要状态

更推荐用“来源卡片 + 状态标签”的形式，而不是一大块文本框。

每个来源卡片建议展示：

- 来源类型
- 条数 / 是否已接入
- 解析状态
- 是否参与本次生成

### 4.4 生成结果概览

大屏不建议直接放完整脑图，而是放“结果摘要 + 可跳转入口”：

- 模块数
- 用例总数
- P0 / P1 / P2 分布
- AI 生成 / 人工补充 占比
- 最近一次生成说明

中间可以放一个“模块分布图”或“精简版结果树”，替代完整脑图。

### 4.5 覆盖与风险概览

这是大屏最有价值的区域，建议强化：

- 总体覆盖率
- 高风险覆盖率
- 未覆盖高风险项数量
- Top 风险点列表
- 推荐补充项数量

风险列表每项建议显示：

- 风险名称
- 所属模块
- 风险来源
- 当前是否已覆盖
- 建议动作

### 4.6 执行与回流概览

大屏里不必直接放执行明细表，而是突出执行结论：

- 最近一次 Jenkins 执行结果
- 成功 / 失败 / 跳过数量
- 最近失败模块
- 是否已发布
- 是否已进入知识库
- 当前质量评分

### 4.7 下一步动作区

这是当前页面最缺的区域，建议单独做一张“行动卡”。

例如自动给出：

- 还缺接口文档，建议补齐后再生成
- 高风险项还有 3 个未覆盖，建议补充测试点
- 已生成但未发布，建议先发布再执行
- 最近执行失败 2 项，建议回看日志并修订用例

这块会让页面从“展示数据”升级为“指导动作”。

## 6. 页面交互建议

大屏不是纯静态页面，建议保留轻量交互：

1. 点击来源卡片，跳转到“输入材料”页签对应区域
2. 点击高风险项，跳转到“生成结果”并高亮对应节点
3. 点击未覆盖项，跳转到“覆盖与风险”的明细区域
4. 点击执行摘要，跳转到“执行与回流”页签
5. 点击“下一步建议”，触发对应动作或过滤条件

这样大屏就会成为四页签工作台的导航入口，而不是一张死面板。

## 7. 视觉风格建议

结合当前项目风格，不建议走“炫酷科技蓝大屏”路线，建议走“业务驾驶舱”风格：

- 背景保持浅色体系，延续当前系统一致性
- 重点用色只给状态和风险，不要全屏大面积紫色
- 摘要卡强调对比度和数字层级
- 风险模块用暖色，执行模块用蓝绿
- 空态页面不要只居中放一句话，要有引导卡和推荐动作

建议视觉关键词：

- 稳定
- 清晰
- 可判断
- 可操作

不建议关键词：

- 炫技
- 过度渐变
- 伪大屏科技感

## 8. 与当前实现的关系

当前 `AICases.tsx` 已经具备可复用基础：

- 顶部摘要卡可作为 KPI 区雏形
- 四页签可保留为二级编辑层
- 覆盖与风险、执行与回流已有骨架
- 生成结果已有脑图和列表双视图

因此更推荐的落地方式不是推翻重做，而是：

1. 在四页签之前增加一个 `总览` 视图
2. 让默认进入页从“结果页”改为“总览页”
3. 把现有顶部 5 张摘要卡升级成更强的 KPI 区
4. 把空态改造成“引导型驾驶舱”

## 9. 分阶段建议

### Phase 1：先做前端总览骨架

建议先落这些内容：

- 总览头部
- KPI 横条
- 输入证据摘要卡
- 结果摘要卡
- 风险摘要卡
- 执行摘要卡
- 下一步建议卡

这一阶段可以主要复用现有前端数据和 mock 数据。

### Phase 2：接真实摘要接口

优先接：

- `GET /workspaces/:id` 扩展摘要字段
- `GET /workspaces/:id/coverage-risk`
- `GET /workspaces/:id/execution-summary`

### Phase 3：补智能动作

后续再做：

- 推荐补充项一键生成
- 风险项跳转联动
- 执行失败项回流
- 质量评分与知识库推荐

## 10. 我建议你们优先确认的 4 个产品问题

1. 这个“大屏”到底是“总览页”，还是“编辑页的大尺寸版本”
2. 首屏默认要强调“生成过程”，还是强调“风险与执行状态”
3. 脑图是否只放到二级页，不在大屏首屏占主位
4. 下一步建议是否要由系统自动生成，而不是只展示静态指标

## 11. 结论

如果目标是把 AI 工作台做成真正可展示、可汇报、可判断的页面，我建议：

- 保留现有四页签作为编辑工作台
- 新增一个“AI 工作台总览大屏”作为默认入口
- 首屏重点展示证据、结果、风险、执行和下一步动作
- 脑图退居二级，不再作为大屏首屏唯一核心

这样更符合你们文档里“从生成页面升级为测试设计中台”的方向。
