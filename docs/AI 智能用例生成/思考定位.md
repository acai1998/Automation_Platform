

**统一产品定位**  
现在的能力更像“AI 根据 PRD 生成脑图用例”。下一阶段建议升级成：

- 输入不再只有 `requirementText`，而是 `需求文档 + 接口文档 + 代码变更 + 历史缺陷 + 生产流量`
- 输出不再只有“脑图”，而是 `风险排序后的测试建议 + 覆盖分析 + 可执行用例 + 执行闭环`
- AI 的角色不只是“写用例”，而是“给出为什么生成这些用例、覆盖了哪些风险、缺了哪些证据”

这会让平台从“生成工具”变成“测试决策平台”。

**建议需求重构**  
建议收敛成 5 个核心模块，先不要一开始铺太大：

1. **证据中心**
   - 接入 PRD、接口文档、历史缺陷、Git diff、Jenkins 构建信息
   - 第二阶段再接生产流量、日志、监控
   - 每次生成都要带“本次使用了哪些输入源”

2. **风险分析引擎**
   - 根据代码变更范围、接口热度、缺陷复发率、业务优先级算风险分
   - 输出高风险模块、推荐优先覆盖场景
   - 这个模块是“为什么生成这些用例”的依据

3. **AI 用例生成工作台**
   - 保留你们现有脑图工作台
   - 但要增加“生成说明”
   - 每条用例要能看到来源：来自需求、来自 diff、来自流量、来自缺陷、来自知识库示例

4. **覆盖率与缺口分析**
   - 不是只看 case 数量
   - 要看需求覆盖率、接口覆盖率、高风险点覆盖率、变更点覆盖率
   - 明确告诉用户“还有哪些高风险区域没覆盖”

5. **执行闭环**
   - 生成后可一键发布为测试集
   - 进入 Jenkins/pytest 执行
   - 执行结果反哺知识库，形成“高质量用例沉淀”

**页面上建议这样改，不要推翻重做**  
现有页面可以继续沿用，但建议把 AI 工作台改成 4 个页签，而不是单一画布：

- `输入材料`
  - PRD、接口文档、附件、变更提交、缺陷单、流量摘要
- `生成结果`
  - 脑图/列表双视图
  - 支持按风险级别、来源、模块筛选
- `覆盖与风险`
  - 显示高风险点、已覆盖/未覆盖、推荐补充项
- `执行与回流`
  - 发布、触发 Jenkins、查看结果、回写质量评分

这样既保住现有实现，又能把“平台感”做出来。

**第三方对接别一口吃太多，MVP 我建议这样排**  
优先级建议是：

1. `P0`
   - Jenkins
   - GitLab/GitHub webhook + diff
   - 现有知识库检索
2. `P1`
   - 缺陷平台（Jira、禅道、Tapd 任选一个）
   - OpenAPI/Swagger 导入
3. `P2`
   - 生产流量
   - ELK/Prometheus
   - Allure/TestRail

原因很简单：当前系统里 Jenkins 和 AI 工作台已经是现成基础，最容易先打通“代码变更 -> 生成 -> 执行”闭环。  
而“生产流量”价值很大，但接入和清洗成本也最高，适合放二期。

**如果按需求文档来写，建议目标这样表述**  
可以直接写成：

- 平台根据需求、代码变更、历史缺陷和生产流量，自动识别高风险区域
- 自动生成高优先级测试用例，并给出覆盖说明与生成依据
- 支持人工审核、编辑、发布、执行和结果回流
- 形成可持续优化的测试知识库，逐步提升生成质量和命中率

**当前最需要先定的 6 个问题**  
这 6 个如果不定，后面很容易需求发散：

1. 你们第一阶段主要生成的是 `接口用例`、`UI 用例`，还是 `pytest 可执行脚本`？
2. 生成结果是“给人编辑确认后再执行”，还是“允许直接自动回归”？
3. 覆盖率口径按什么算：需求点、接口、代码变更点，还是业务流程？
4. 生产流量你们实际能拿到什么：API 调用、用户路径、异常日志，还是只有聚合报表？
5. 缺陷平台准备用哪一个，字段是否能稳定拿到？
6. AI 生成的最终落点是“脑图工作区”，还是要同步落到现有 `TestCase/Task/Jenkins` 执行链路？

建议是先做一个真正能落地的 `MVP 版需求`：  
`需求/接口文档 + Git diff + 知识库检索 + Jenkins 执行回流`，先把闭环打通；生产流量和缺陷平台放二期。
